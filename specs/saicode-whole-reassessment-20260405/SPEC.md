# saicode 全仓重新审视 Spec

## Goal

重新审视当前整个 `saicode` 的真实健康状态，判断“还剩多少问题”，并把问题按严重度分层，而不是只围绕最近修过的单点做局部结论。

## Scope

### In scope

- 重新检查以下方面：
  - 启动与非交互 CLI 健康度
  - 工具调用与搜索链路是否真实可用
  - 默认模型与运行时配置是否一致
  - 自动化门禁是否可信
  - 文档、脚本、代码当前行为是否一致
  - 代码结构层面的明显剩余风险
- 产出：
  - 问题分级清单
  - 每项问题的证据
  - 当前是否建议继续同轮修复的判断

### Out of scope

- 一次性重写整个架构
- 无证据的大规模性能臆测
- 为了“看起来更干净”而做无关代码整理

## Constraints

- 必须以真实命令、真实探针和当前代码为准
- 不回退用户已有脏工作区改动
- 问题要区分为：
  - 已影响可用性的问题
  - 高概率后续出问题的结构性风险
  - 纯技术债

## Acceptance

1. 至少完成一轮仓库级健康检查与关键 CLI/工具真实探针。
2. 至少完成一轮源码结构扫描，指出当前最关键的结构性风险。
3. 最终给出“还剩多少问题”的分级清单，而不是笼统评价。
4. 说明哪些已经是阻塞问题，哪些只是后续应收敛的债务。
