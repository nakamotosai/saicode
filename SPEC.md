# saicode hardening Spec

## Goal

把当前 `saicode` 从“表面能启动、但首请求不稳、配置口径分裂、缺少回归门”的状态，收敛到“本机真实请求可跑通、配置与文档一致、最小回归可重复、常用入口更轻”的状态。

在此基础上，本轮继续把一次性任务链路收敛到“完整 CLI 默认也能跑、工具调用可实测、Telegram 包装输入不串味、常见一次性任务尽量少烧上下文”的状态。

本轮体检新增确认一项真实运行约束：当前 `cliproxyapi` 上的 `qwen/qwen3.5-397b-a17b` 不适合作为本机默认非交互模型，因此默认值必须切回已 live probe 验证可用的 `gpt-5.4` 路线。

## Scope

### In scope

- 统一 `cpa` provider 配置口径：
  - 环境变量
  - 错误提示
  - README / `.env.example`
  - 运行时读取顺序
- 打通本机首请求主链路：
  - `saicode -p "Reply with exactly: ok"`
  - 使用当前真实统一入口
- 补最小回归门：
  - 至少覆盖模型别名/配置解析
  - 至少覆盖 `cpa` 配置口径不会再漂移
- 轻量提速收口：
  - 优先优化常用非交互入口
  - 不追求一次性重写整个启动链
- 完整 CLI 一次性任务稳定性：
  - 默认模型不再误回退到旧的 `claude-sonnet-4-6`
  - 默认模型不再落到当前已知会卡死的 Qwen 397B 路线
  - `stream-json` / 指定工具集不再被错误送进 recovery 快路径
  - 一次性 print 请求默认可进入 lean context，但不能误伤常用本地工具
- 一次性任务真实探针：
  - 纯文本
  - Read / Grep / Write / Edit
  - Telegram 工程桥包装输入
- 保持当前 `cpa/...` 模型面与旧别名兼容

### Out of scope

- 全仓历史残留一次性清零
- 全面重构 `main.tsx`
- 恢复旧的 Claude/Anthropic 产品旁支
- 新增复杂的新 provider 架构

## Constraints

- 不把密钥硬编码进仓库文件
- 不伪造“类型全绿 = 稳定可用”
- 优先复用当前机器已经验证可用的统一入口
- 先拿真实请求和最小回归过闸，再继续做更深层清理
- 不做会破坏当前 `cpa/...` 模型面的回退

## Acceptance

至少满足以下条件才算这一轮修复达标：

1. `saicode --help` 可正常输出
2. `saicode -p "Reply with exactly: ok"` 在本机真实返回 `ok`
3. `cpa` 相关错误提示与真实读取口径一致，不再误导到错误变量名
4. 本机存在持久可复用的 `saicode` provider 配置来源，不需要每次手工临时注入
5. 至少一组自动化验证可跑，能覆盖本轮修复的核心配置逻辑
6. 常用非交互入口的冷启动指标较当前基线有可见改善，或至少去掉最明显的无谓绕路
7. 不显式传 `--model` 的完整 CLI 一次性请求也能在本机走通
8. `Read` / `Grep` / `Write` / `Edit` 的一次性任务至少各有一条真实探针通过，且工具调用与任务意图一致
9. Telegram 工程桥包装输入可直接得到正确简短结果，且不发生无谓工具调用
10. 一次性 lean context 的系统提示显著缩小，避免简单任务继续背整套重上下文

## Risks

- 当前问题不是单点 bug，而是“配置、文档、运行时、验证”多面漂移
- 若统一入口配置源判断错了，会把 `saicode` 接到旧密钥或错误路由
- 若继续把不稳定模型留作默认值，`saicode -p` 会在“帮助页和测试都绿”的情况下继续假健康
- 启动提速很可能只能先拿到一轮局部收益，无法一轮吃掉所有历史包袱
